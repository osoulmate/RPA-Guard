# MBT-Engine: 模型驱动自动化测试路径生成平台

**MBT-Engine** 是一款集系统架构建模、实时可视化渲染与测试路径自动生成于一体的轻量级测试工具平台。它基于 **Model-Based Testing (基于模型的测试)** 理论，通过解析 Mermaid 拓扑结构，利用带有防环机制的图遍历算法，实现从“业务架构”到“全路径测试用例”的自动化转化。

---

## 🌟 核心特性

* **可视化建模**：集成 Mermaid.js，支持实时预览 CDFD（控制数据流图）及各类系统架构。
* **智能路径穷举**：后端采用深度优先搜索（DFS）算法，自动发现系统中所有可能的端到端业务路径。
* **闭环/防环机制**：针对数据库读写、循环重试等复杂逻辑，具备智能节点计数与死循环熔断功能。
* **多场景适配**：内置金融 ATM、电商交易、智能安防（IoT）三大行业标准模型示例。
* **全屏审查模式**：支持架构图一键全屏，便于复杂系统的详细评审。
* **测试资产导出**：支持一键生成符合 Excel 标准（带 BOM 头）的 CSV 测试用例表。
* **安全访问控制**：内置极客风登录拦截器，保护企业内部架构模型资产。

---

## 🛠️ 技术栈

### 前端 (Frontend)

* **框架**: React 18 + Vite
* **语言**: TypeScript
* **渲染**: Mermaid.js (v11+)
* **布局**: CSS Flexbox + 暗黑模式 UI

### 后端 (Backend)

* **框架**: FastAPI (Python)
* **算法**: Recursive DFS with Path Tracking
* **解析**: 正则表达式宽容模式 (支持自适应连线长度)

## 🚀 快速开始

### 环境要求

* Node.js (v16+)
* Python (3.10+)

### 1. 克隆与初始化

```bash
git clone https://github.com/osoulmate/mbt-engine.git
cd mbt-engine

```

### 2. 后端部署 (FastAPI)

```bash
# 进入后端目录 (假设在 src 目录下)
cd src
# 安装依赖
pip install fastapi uvicorn pydantic
# 启动服务
uvicorn main:app --reload --port 8000

```

### 3. 前端部署 (React)

```bash
# 回到项目根目录
cd front/
npm install
# 启动开发服务器
npm run dev

```

打开浏览器访问 `http://localhost:5173`。

## 🧩 核心逻辑说明：防环机制

本系统在解析 `Withdraw -> account_file -> Check_Password` 这种典型的 CDFD 数据闭环时，采用了“路径局部访问计数”策略。当算法检测到下一个目标节点已存在于当前路径栈中时，会自动将其标记为 `(闭环/终点)` 并停止递归，从而完美避开了图遍历中的死循环陷阱，同时完整记录了状态写入这一业务终态。

## Q&A

### Q1：系统是如何将 Mermaid 文本转化为计算机可读的图结构（Graph）的？

**🔴 难点剖析**：Mermaid 语法非常灵活，用户为了调整视觉排版，可能会混用 `-->`、`--->` 甚至 `----`，传统的字符串分割法会瞬间失效。

**🟢 回答**：
“我们没有采用脆弱的字符串切割，而是**在 Python 后端自研了一套宽容度极高的正则表达式解析器**。
例如，针对带条件的连线，我使用了 `re.compile(r'([A-Za-z0-9_]+)\s*-+\s*"([^"]+)"\s*-+>\s*([A-Za-z0-9_]+)')`。这里的核心是 `-+`（匹配一个或多个破折号），它使得解析器能够完美兼容用户为了排版而添加的长线（如 `--->`）。同时，解析器会自动提取节点别名与图边关系，在内存中实时构建出高保真的**邻接表（Adjacency List）**数据结构。”

### Q2：在生成测试路径时，如何解决业务逻辑中的“数据环（死循环）”问题？

**🔴 难点剖析**：在 ATM 系统中，取款（Withdraw）后需要更新数据库（account_file），而后续操作可能再次校验密码，形成 `P3 -> D1 -> P2` 的物理死循环。标准的图遍历算法会遭遇内存溢出（Stack Overflow）。

**🟢 回答**：
“我们在深度优先搜索（DFS）算法中引入了**‘基于局部路径的环路检测（Cycle Detection）与熔断机制’**。
传统的全局 `visited` 集合不适用于多路径生成，因为合法的业务流允许在不同用例中重复访问节点。因此，我们的算法只检查**‘当前正在遍历的这一条路径栈’**中是否出现了重复节点。一旦检测到（例如数据写回操作导致成环），算法会将其智能标记为 `(闭环/终点)` 并主动切断递归。这不仅防止了服务崩溃，还完美记录了‘状态回写’这一核心的业务终态。”

### Q3：为什么选择 React (前端) + FastAPI (后端) 的分离架构，而不是用纯前端 JS 完成所有事？

**🔴 难点剖析**：这是一个经典的架构选型问题。虽然 JS 也能跑 DFS 算法，但将所有逻辑耦合在前端会导致平台难以扩展。

**🟢 回答**：
“这主要是基于**关注点分离（Separation of Concerns）**和**未来演进**的考量：

1. **前端专注渲染**：React 配合 Mermaid.js 需要处理复杂的 DOM 操作和 SVG 实时重绘，并且要管理多场景状态与全屏交互，保持前端轻量能保证极佳的用户体验。
2. **后端专注算法与扩展**：Python（FastAPI）在处理复杂数据结构和图算法时更加优雅且高效。更重要的是，Python 是人工智能生态的绝对标准语言。将核心解析引擎放在后端，意味着我们未来可以极其方便地接入各种大语言模型（LLM）或 AI 代理框架，实现更高阶的自动化。”

### Q4：在 CDFD 模型中，系统是如何区分“控制流”和“数据流”的？

**🔴 难点剖析**：图中既有代表操作步骤的连线（控制流），也有代表读取数据的连线（如 `account_file` 到 `Check_Password`），如果不加区分，生成的测试用例会逻辑混乱。

**🟢 回答**：
“我们采用了一种**‘视觉语法与算法正则双解耦’**的精妙设计。
在前端建模时，我们规定‘纯数据读取’使用虚线（`-.->`）绘制，以保证架构图视觉上的完整性。而在 Python 后端，我们的核心正则表达式被严格限制为只捕获实线（`-->` 等）。
这样一来，系统在物理层面直接过滤掉了非执行性质的数据流，确保图遍历算法只沿着真实的**业务控制控制流**推进；同时，对于没有入度只有出度的实线起点（如 `card_id`, `amount`），算法会自动将其识别为**前置测试数据**，从而生成完全符合软件测试规范的端到端用例。”
